## 0. 使用与维护规则

新对话接手项目时，先完整阅读本文件，再核对真实源码、Git、微信开发者工具和云端状态。

事实优先级：

```text
真实源码 / Git / 云端查询 / 控制台实际配置
> 本总交接文档的“当前状态”
> 最新阶段正式文档和 README
> 旧的编号交接文档
```

旧交接文档记录的是各阶段当时的快照，其中“尚未部署”“禁止提交”“等待人工验收”等结论可能已经被后续阶段覆盖，不能脱离时间背景直接执行。

从本文件创建后开始：

1. 后续上下文接近上限时，直接更新本文件，不再另建新的编号交接文档，除非用户明确要求。
2. 当前状态发生变化时，应修改对应章节，不能只在文末追加一个与正文矛盾的新结论。
3. 旧的 `04.md`—`24.md` 作为历史资料保留，不擅自删除、改写、暂存或提交。
4. 本文件是否提交由用户决定；没有明确 Git 授权时只更新工作区文件。
5. 任何“已部署”“已验收”“已上线”结论都必须有真实证据，自动测试不能代替人工或双账号验收。

## 4. 当前核心架构

### 4.1 分层原则

```text
Page / Component
→ Service
→ wx.cloud.callFunction()
→ Cloud Function
→ Database / Cloud Storage
```

页面负责展示和交互；Service 负责参数标准化、超时和错误映射；云函数负责身份、权限、可信字段、事务和数据库访问。

客户端不直接访问 `users`、`products` 或 `favorites` 集合。

### 4.2 认证链路

```text
App 启动
→ AuthStore.bootstrap()
→ AuthService
→ authUser/current
→ 云端校准状态
```

主动登录：

```text
受保护入口
→ AuthGuard 白名单目标
→ 登录页
→ AuthStore.login()
→ authUser/login
→ 安全用户摘要
→ 返回原目标
```

### 4.3 商品读取链路

```text
pages/home 或 pages/product-detail
→ ProductService
→ productQuery(list/detail)
→ products
```

本人商品：

```text
pages/my-products
→ MyProductsService
→ productQuery(myProducts)
→ getWXContext().OPENID
→ products
```

### 4.4 发布链路

```text
pages/publish
→ ProductPublishService
→ 云存储上传
→ createProduct
→ users 身份记录
→ products 幂等写入
→ 商品详情 / 首页刷新
```

### 4.5 编辑与状态管理链路

```text
pages/my-products 或 pages/product-edit
→ MyProductsService / ProductEditService
→ manageProduct
→ 服务端所有权、状态和 version 校验
→ products 事务更新
→ 必要时服务端清理旧图片
```

### 4.6 收藏链路

```text
商品详情 / 我的收藏
→ FavoriteService
→ favoriteProduct
→ favorites + products.favoriteCount 事务
```

### 4.7 公开主页链路

```text
pages/user-profile
→ PublicUserService
→ userQuery(publicProfile/publicProducts)
→ users + products
```

### 4.8 消息链路

```text
商品详情 / 消息中心 / 聊天页
→ MessageService
→ CloudService.ensureCloudReady()
→ messageAction / messageQuery
→ conversations + messages + users + products
```

写消息使用数据库事务；会话和消息分页都使用“时间 DESC + `_id` DESC”双字段游标。

所有客户端云函数和云存储 Service 统一复用 `services/cloud-service.js`。`App.onLaunch` 的当前顺序为 `AppStore.initialize()` → `CloudService.ensureCloudReady()` → `AuthStore.bootstrap()`；业务调用仍会在自身入口显式等待同一个初始化 Promise，因此不依赖 `onLaunch` 异步返回值，也不使用 `setTimeout` 猜测云能力是否就绪。初始化成功后复用 ready 状态，失败后清空 Promise 以允许安全重试。

## 7. 数据模型、状态和索引

### 7.1 users

稳定用户 ID：

```text
u_ + SHA-256(APPID:OPENID) 的前 32 个十六进制字符
```

当前继续将 `users._id` 用作安全的：

```text
publicUserId
```

最终服务端内部身份字段继续统一使用：

```text
openid
```

客户端不能提交、缓存或接收该字段。用户资料模型为：

```text
_id / openid
nickname / avatarUrl / campus / bio
profileCompleted / status / role
createdAt / updatedAt / lastLoginAt
```

用户 `_id` 由 `APPID:OPENID` 的 SHA-256 摘要确定，同一真实身份的首次、重复和并发登录只能落到同一文档，因此本任务没有新增 `openid` 唯一索引。云端 `users` 当前只有系统 `_id`、`_openid` 索引；不要为满足文档形式重复建立等价索引。

公开资料只允许白名单字段：

```text

nickname
avatarUrl
campus
bio
joinDate
activeProductCount
```

OPENID 不能作为 URL 参数、公开响应字段或客户端缓存字段。

## 8. 必须保持的安全与一致性边界

1. 客户端不得直接读写正式数据库集合。
2. 客户端不得传入、决定、缓存或公开 OPENID。
3. 云函数必须通过 `cloud.getWXContext()` 获取可信微信身份。
4. `sellerOpenid`、`userOpenid` 只能作为服务端内部字段。
5. 所有商品写操作必须在云端校验所有权。
6. `manageProduct` 必须保留字段白名单、数据库事务和 `version` 并发控制。
7. 商品删除保持软删除，不执行客户端可控的数据库物理删除。
8. 旧图片清理由服务端根据事务后的可信商品状态计算，不能信任客户端删除清单。
9. 数据库事务与云存储删除不是跨服务原子操作；清理失败不能回滚已成功的商品事务。
10. 收藏关系与 `favoriteCount` 必须在同一事务内保持一致。
11. 收藏操作必须保持幂等，计数不能重复增加或小于 0。
12. 公开用户资料和商品数据必须显式白名单映射。
13. 公开列表、详情、公开主页和收藏列表都必须正确隔离 `deleted`。
14. 不恢复 `seed`、debug、Mock 写正式数据库入口。
15. 不为通过测试而降低数据库或云存储权限。
16. 不向仓库、日志或交接文档写入 AppSecret、Access Token、永久云密钥、测试账号、完整 OPENID 或完整敏感 fileID。
17. 公开浏览不能被认证临时失败阻断。
18. 路由回跳只接受固定白名单目标，不能信任任意页面参数。
19. `conversations` 和 `messages` 必须保持客户端不可直接读写。
20. 创建会话时客户端只能提交 `productId`；卖家、双方身份与槽位必须由云端计算。
21. 查询会话、查询消息、发送和标记已读都必须校验参与者。
22. 消息重试必须沿用原 `clientMessageId`，不能重复增加接收方未读数。

## 13. 开发、部署和 Git 操作原则

1. 保留微信原生小程序架构，不无理由迁移框架。
2. 优先扩展现有 Service 和云函数 action，避免重复数据体系。
3. 修改范围保持小而可验证，不能覆盖用户已有本地修改。
4. 每次阶段报告分别说明：
   - 源码是否完成；
   - 自动验证是否通过；
   - 微信开发者工具是否编译；
   - 云函数是否真实部署；
   - 云端配置是否核对；
   - 人工验收哪些通过、哪些未通过。
5. 未经用户明确授权，不执行 commit、push、打标签、强制推送或生产破坏性写入。
6. 暂存时使用显式文件列表，避免 `git add .` 把交接资料加入提交。
7. 不执行 `git reset --hard`、`git clean` 或批量恢复工作区。
8. 只部署实际修改或新增的云函数。
9. 部署失败时先查询真实云端状态，再决定是否安全重试。
10. 云函数代码部署、函数配置修改、数据库权限修改、索引操作和生产数据写入是不同权限边界，不互相推定授权。
