> 更新日期：2026-07-24
> 云环境：`cloud1-d9gpdpv6p2db56d8e`
> 状态：源码、自动验证、开发者工具编译、云函数部署、云端反查和双账号人工验收均已完成；仅第三个真实账号的非参与者越权真机测试低风险延期

## 1. 最终认证链路

```text
小程序启动
→ AuthStore.bootstrap()
→ authUser/current
→ cloud.getWXContext() 获取当前真实微信身份
→ 确定性 users._id 查询
→ 已完善资料则恢复会话
→ 未注册或资料未完善则进入登录资料页
```

首次登录：

```text
用户主动选择头像、填写昵称
→ authUser/login 创建或复用当前真实身份用户
→ AvatarService 校验并上传头像到当前用户专属目录
→ authUser/updateProfile 仅更新当前用户安全资料
→ AuthStore 缓存安全摘要
→ AuthGuard 按白名单返回原目标
```

身份只来自云函数上下文。客户端不提交、不缓存、不接收 OPENID。

## 2. 双账号验收前准备

1. 准备两个不同微信账号：
   - 账号 A：卖家；
   - 账号 B：买家。
2. 使用以下任一组合：
   - 推荐：两部手机安装同一体验版；
   - 可用：微信开发者工具登录账号 A，一部手机登录账号 B。
3. 打开云开发控制台，准备只读观察：
   - `users`；
   - `products`；
   - `conversations`；
   - `messages`；
   - `authUser`、`createProduct`、`messageAction`、`messageQuery` 日志。
4. 不使用 `product-009` 等旧种子商品。必须由账号 A 在新登录流程完成后重新发布商品。
5. 记录验收前集合数量，不记录或传播完整 OPENID。

## 3. 登录与用户唯一性

### 账号 A

1. 清除账号 A 的小程序本地缓存。
2. 从“我的”或发布入口进入登录页。
3. 主动选择头像、填写昵称，可选填写校园信息。
4. 点击“确认并登录”，确认 Loading 能关闭且成功返回原目标。
5. 在数据库控制台确认：
   - 账号 A 对应 `users` 文档只有一条；
   - `nickname`、`avatarUrl` 是账号 A 主动提交的资料；
   - `profileCompleted=true`；
   - `_id` 为稳定安全公开 ID；
   - 内部身份字段存在，但不复制到验收记录。
6. 退出登录，重新进入小程序，确认恢复同一用户。
7. 再次登录，确认没有新增第二条用户记录，公开用户 ID 不变。

### 账号 B

按账号 A 的步骤完成账号 B 登录，并确认：

- 两个账号对应两条不同用户记录；
- 两个公开用户 ID 不同；
- 两个账号的昵称、头像和本地缓存互不串用。

设备要求：两部手机，或开发者工具加一部手机。
观察要求：数据库控制台查看用户数量和安全公开字段；必要时查看 `authUser` 日志，但不得记录完整 OPENID。

## 4. 商品与本人识别

1. 账号 A 发布一个全新的测试商品。
2. 数据库控制台确认商品 `sellerId` 对应用户 A，内部卖家身份由云函数生成。
3. 账号 A 打开该商品：
   - 页面稳定识别为本人商品；
   - 点击联系卖家返回“不能给自己发送私信”；
   - 不应显示“商品不存在”。
4. 账号 B 打开同一商品，页面不得识别为本人商品。

设备要求：两部手机，或开发者工具加一部手机。
观察要求：数据库控制台只读核对商品卖家映射；云函数日志核对错误码。

## 5. 会话、收发和未读

1. 账号 B 点击“联系卖家”：
   - 首次点击创建一条会话；
   - 重复点击复用同一会话；
   - `conversations` 不增加第二条相同商品双方会话。
2. B 向 A 发送一条文本消息：
   - B 侧气泡方向为自己；
   - A 的会话列表未读数增加 1；
   - 最后消息摘要正确。
3. A 进入聊天：
   - A 的当前会话未读清零；
   - B 的未读槽位不被误清零。
4. A 回复 B：
   - A 侧气泡方向为自己；
   - B 收到新消息并出现未读数。
5. 双方退出聊天再进入，历史消息仍存在且顺序正确。
6. 将小程序切入后台，确认轮询停止；回到前台后轮询恢复。

设备要求：推荐两部手机；体验版最接近最终环境。
观察要求：数据库控制台观察会话、消息数量和未读槽位；云函数日志观察 `messageAction` 与 `messageQuery` 结果。

## 6. 幂等、越权和状态规则

1. 使用调试工具对同一 `clientMessageId` 重试发送，确认只存在一条消息且未读只增加一次。
2. 使用第三个真实账号 C 或受控测试调用：
   - 读取 A/B 会话应返回 `FORBIDDEN`；
   - 读取 A/B 消息应返回 `FORBIDDEN`；
   - 向 A/B 会话发送应返回 `FORBIDDEN`。
3. 在既有会话建立后依次验证商品状态：
   - `offline`：不能新建会话，既有会话可继续；
   - `sold`：不能新建会话，既有会话可继续；
   - `deleted`：保留历史只读，不允许继续发送。
4. 多端同时发送不同消息，确认消息不丢失、会话摘要最终一致。
5. 检查两个账号的用户资料、商品、收藏、会话和未读数据互相隔离。

设备要求：三账号越权最好使用体验版加两部手机或额外测试人员。
观察要求：数据库控制台和云函数日志。禁止直接修改参与者字段伪造通过结果。

## 7. 验收后清理

- 不删除任何真实 `users` 文档。
- 测试商品按正常商品生命周期软删除，不直接物理删除。
- 会话和消息是否保留由发布方决定；清理前先确认引用关系并形成独立迁移方案。
- 不删除仍被用户资料引用的头像文件。
- 不把完整 OPENID、fileID、测试账号或密钥写入文档、截图或 Git。
